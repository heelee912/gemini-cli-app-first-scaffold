# Empirical Build-Like Drivers

These notes capture what separated stronger local runs from weaker ones. They are operating drivers, not visual references.

## Keep

- Treat the Vite app as the source artifact. Do not reduce the task to a chat answer or handcrafted static document when the goal is Build-grade output.
- Let the user brief, supplied assets, supplied references, and artifact purpose determine the concrete visual system.
- Build artifact-specific structure rather than a generic page.
- Use named data, state, selection, comparison, inspection, export, navigation, or mode controls when those behaviors genuinely help the artifact.
- Keep critical content readable in the verified stable render. Motion and reveal patterns are valid when they improve the artifact, but the preview must not remain blank or unusable after normal startup and animation settling time.
- Design mobile behavior from the beginning. Dense controls and data surfaces need explicit narrow-screen behavior.
- Preserve a strong visual skeleton during repair. Fix concrete defects without flattening a promising direction into a safer generic template.

## Avoid

- Do not copy a prior artifact's palette, motif, layout, domain content, copy, component names, or styling.
- Do not turn heuristic source-depth metrics into arbitrary design goals.
- Do not add controls only to satisfy a count. Controls must change what the user can inspect, compare, export, navigate, or decide.
- Do not use the scaffold as a house style. The scaffold exists to provide the execution environment and source prompt context only.
